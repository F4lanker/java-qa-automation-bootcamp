BaseApiTest.requestSpec — почему protected?
 Модификатор доступа позволяет видеть метод в классках наследниках и внутри пакета

Что будет если сделать private?
Тогда доступ будет только непосредственно из класса

Что будет если сделать public?
Слишком много доступа, внешние классы не должны видеть метод
Почему protected — оптимальный выбор?
Руководствуясь золотым правилом  Используй **самый ограниченный** модификатор, который позволяет коду работать. Даем самый минимальный доступ для адекватной работы.


BaseApiTest.setup() — почему public?

Может ли быть protected? Метод используется Junit, поэтому метод должен быть public. 
Почему JUnit требует public (или package-private)? Иначе у Junit не будет доступа, и тест упадет.

DTO поля — почему private? DTO - макетный класс, для доступа и изменения используются конструкторы, сеттеры и геттеры

Что даёт инкапсуляция в DTO? - Инкапсуляция насколько я понимаю, это упаковкаправ доступа к полям класса, через геттеры и сеттеры, конструкторы. Дает возможность контролировать значения, через валидацию, либо дать только право на чтение(есть геттеры, но нет сеттеров)
Может ли быть оправдано public поле в DTO? - Тогда мы потеряем преимущества работы через инкапсуляцию. Можно будет создать и присвоить значение минуя конструктор, геттер и сеттер, а именно напрямую
